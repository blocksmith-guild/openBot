# TOOL: reversible removal (remove)

## INTENT
Safely and reversibly remove a file from the active workspace without permanent data loss.

## STRATEGY (The "Safety-Net" Phase)
1.  **RCS Check-in**:
    - If the file is tracked by RCS (associated `,v` file exists), check in the file first: `ci -u -m"De-materializing: [reason]" <file>`.
    - If no `,v` file exists but tracking is desired before removal, initialize with `ci -i -t-"De-materializing: [reason]" <file>`.
2.  **Trash/Archive**:
    - If a `.trash` directory exists in the project root:
        - Move the file to the `.trash` folder: `mv <file> .trash/`.
        - If a conflict exists (file already exists in `.trash`), use a versioned name: `<file>~1`, `<file>~2`, etc.
    - If no `.trash` exists, use `backup.pl -f <file>` to create a versioned backup in the current directory (e.g., `<file>~1`).
3.  **Git Update**: If the file is tracked by Git, perform a "deletion record": `git rm <file>`.
4.  **Audit Trail**: Update any relevant `.lof` (List of Files) or `README.md` to reflect the removal.

## MANDATE
- **Zero Destruction**: Never use `rm -rf` on a file. Always preserve the signal in the `.trash` or as a versioned backup.
- **De-materialization**: Treat removal as a transformation from "Signal" to "History".
- **Conflict Resolution**: Use `*~n` where `n` is an incrementing integer (1, 2, 3...) for filename conflicts in the `.trash` folder.

---
*"Nothing is ever truly gone; it just becomes a different kind of signal."*
